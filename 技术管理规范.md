# 技术管理规范 （部门试行版）

> 本文件为数字业务部技术团队的工作规范。
> 目标仓库：GitHub Organization `.github` repo
> 最近更新：2026-03-11

---

## 适用范围

本规范适用于涉及生产环境部署或工期超过 3 个工作日的技术项目。紧急修复和微调可走简化流程，但须事后补录。

---

## 一、技术职责边界

技术团队接到需求后，以下工作首先属于技术职责范畴：

1. 调研需求的技术可行性
2. 确定实施方案和开发流程
3. 编写开发方案文档，列出需要产品确认的要点
4. 返回给产品做确认，获得明确确认后再动手

原则：调研实施策略和确定开发方案首先是技术的职责；产品为辅，主要负责确认和验收开发方案。

## 二、文档能力要求

写文档之前先想清楚两件事：

- 这个文档的目的是什么
- 阅读对象是谁

好文档的共同点：

1. **简明扼要**——不能太长
2. **重点突出**——核心结论放在最前面
3. **列确认点**——内部交流文档须显式列出需要交付对象确认的要点

善用 AI 辅助和网络查询，建立对不同类型文档最佳实践的认知。

## 三、项目生命周期

技术接需求后，项目按以下阶段推进。每个阶段有明确的交付物和审核条件，未通过审核不得进入下一阶段。

### 阶段一：方案（开发前）

| 工作 | 说明 |
| --- | --- |
| 调研 | 理解需求背景，调研技术方案 |
| 确认关键要素 | 列出技术选型、影响面、风险点，与产品/主管确认 |
| 拆分任务 | 在任务管理平台上建立任务并分解为可执行的子任务 |
| 协调时间 | 与相关人员确认时间窗口和优先级 |

**交付物**：方案文档（技术选型、影响面、风险预案、确认点）+ 实施计划（任务分解、时间节点、责任人）

**审核条件**：方案文档完成 → 主管/产品确认 → 方可进入开发

### 阶段二：执行（上线前）

| 工作 | 说明 |
| --- | --- |
| 开发实施 | 按确认的方案执行，遇到偏差及时上报 |
| 文档维护 | 方案文档、实施记录持续更新 |
| 测试验证 | 按验收标准完成测试 |

**交付物**：测试/验收记录（测试范围、验收标准、通过情况）

**审核条件**：开发完成 → 测试验证 + 影响面确认 + 回滚预案就绪 → 方可上线

### 阶段三：收尾（关闭前）

| 工作 | 说明 |
| --- | --- |
| 复盘 | 总结经验，沉淀可复用的知识文档 |
| 文档归档 | 确认四份必备文档齐全并归档 |
| 遗留问题清零 | 确认无未关闭的遗留问题 |

**交付物**：复盘总结（经验教训、可复用资产、遗留问题）

**审核条件**：复盘完成 + 文档归档 + 遗留问题清零 → 方可关闭项目

### 四份必备文档

每个技术项目须在生命周期中产出并维护以下四份文档：

1. **方案文档**：技术选型、影响面、风险预案、确认点
2. **实施计划**：任务分解、时间节点、责任人
3. **测试/验收记录**：测试范围、验收标准、通过情况
4. **复盘总结**：经验教训、可复用资产、遗留问题

> 以上机制从 MySQL 升级项目中沉淀而来，后续将在更多项目中验证和完善。

## 四、敏感信息管理

**禁止将以下内容提交到任何 Git 仓库**（包括私有仓库）：

- API 密钥、Token、密码
- `.env` 及任何包含环境变量的配置文件
- 数据库连接字符串
- 第三方服务凭证（微信、支付宝、阿里云等）

**执行要求**：

1. 每个项目根目录须有 `.gitignore`，覆盖上述文件类型
2. 敏感配置统一通过环境变量注入，不硬编码在源码中
3. 若发现历史 commit 中包含敏感信息，须立即上报并处理（使用 `git filter-repo` 清理历史 + 立即轮换对应密钥）

> 敏感信息一旦推送到远程，即使删除 commit 也可能已被缓存或爬取，**唯一安全的做法是立即轮换密钥**。

## 五、AI 工具使用责任

AI 是工具，为 AI 执行结果负责的仍然是人。

- AI 产出的代码、文档、方案，必须由负责人审查后才能采纳
- 不能以"AI 生成的"作为质量问题的免责理由
- 鼓励使用 AI 提升效率，但最终交付质量由人负责

## 六、issue 自主提交

工程师应自主将工作中发现的问题提交到任务管理平台（如 ones）。

两层意义：

1. **建立 issue 的过程是调研和理解问题的过程**——与 TDD 共享类似的原理：先定义问题（写测试），再解决问题（写代码）
2. **工程师身份的转变要求学会"提需求"**——要让和你合作的人类伙伴看得懂，更要让和你合作的机器伙伴看得懂，因为工程师工作的很大一部分内容是指挥机器。
